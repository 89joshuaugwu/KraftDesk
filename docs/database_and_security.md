# KraftDesk — Data Model & Database Security

This document covers our database schemas, records relationships, and security policies. It explains how KraftDesk structures and protects data in Google Cloud Firestore.

---

## 1. NoSQL Schema Overview

KraftDesk uses **Google Cloud Firestore**, a NoSQL document database, for all dynamic structured data. Data is stored in **Documents** that reside in **Collections**. Some documents also contain **Subcollections** for child records (like version histories and comments on a specific poster).

Here is a visual map of the data structure hierarchy:

```text
Firestore Root
 ├─ users (collection)
 │   └─ {uid} (user document)
 ├─ categories (collection)
 │   └─ {categoryId} (category document)
 ├─ posters (collection)
 │   └─ {posterId} (poster document)
 │       ├─ versions (subcollection)
 │       │   └─ {versionNumber} (version document)
 │       └─ comments (subcollection)
 │           └─ {commentId} (comment document)
 └─ notifications (collection)
     └─ {uid} (user's notification inbox)
         └─ items (subcollection)
             └─ {notifId} (individual notification)
```

---

## 2. Collections and Document Schemas

### 1. User Records (`/users/{uid}`)
Stores account profile information, user roles, and personal notification preferences.

* **Path**: `/users/{uid}` (where `{uid}` is their unique identifier generated by Firebase Auth)
* **Fields**:
  - `uid` (string): Same as the document ID.
  - `email` (string): Account email address.
  - `displayName` (string): Full name or handle displayed on comments/posters.
  - `avatarUrl` (string): Link to their profile picture.
  - `role` (string): Their permission tier. Valid options: `"designer" | "reviewer" | "admin"`.
  - `notificationPrefs` (map):
    - `newSubmission` (boolean): Notify if a designer uploads a poster (intended for reviewers/admins).
    - `statusChange` (boolean): Notify when a poster's status is modified (intended for designers).
    - `newComment` (boolean): Notify when someone replies.
  - `createdAt` (timestamp): Record creation date.

---

### 2. Categories (`/categories/{categoryId}`)
Lists the design categories (e.g. "Marketing", "Event", "Banner") posters belong to.

* **Path**: `/categories/{categoryId}`
* **Fields**:
  - `name` (string): The category label.
  - `posterCount` (number): Number of posters active under this category (used to block deletion if > 0).
  - `createdBy` (string): UID of the admin who created it.
  - `createdAt` (number): Creation timestamp.

---

### 3. Posters (`/posters/{posterId}`)
The primary records in KraftDesk. Represents a design project and contains state and URL pointers.

* **Path**: `/posters/{posterId}`
* **Fields**:
  - `title` (string): The title of the poster.
  - `category` (string): Category ID reference pointing to a document in `/categories`.
  - `tags` (array of strings): Descriptors for search filters.
  - `status` (string): Current workflow stage. Valid options: `"draft" | "pending" | "approved" | "changes_requested" | "rejected" | "published"`.
  - `currentVersion` (number): Counter representing the latest revision (e.g. version 1, 2, 3).
  - `cloudinaryPublicId` (string): Unique image identifier inside Cloudinary.
  - `secureUrl` (string): Direct, clean (un-watermarked) URL of the source image file on Cloudinary. **Important**: This field is kept secure and never exposed directly in image tag properties.
  - `previewUrl` (string): A public-friendly URL that Cloudinary processes. For pending or changed posters, this URL instructs Cloudinary to overlay text watermarks dynamically.
  - `uploadedBy` (string): UID of the designer who uploaded the poster.
  - `reviewedBy` (string | null): UID of the reviewer/admin who last changed the status.
  - `createdAt` (number): Original upload date.
  - `updatedAt` (number): Timestamp of the last revision or review update.
  - `publishedAt` (number, optional): Timestamp of when it was published.

---

### 4. Poster Subcollections

#### A. Revisions (`/posters/{posterId}/versions/{versionNumber}`)
Tracks historical versions. If a reviewer requests changes, the designer uploads a new version. The old versions remain viewable.
* **Fields**:
  - `versionNumber` (number): Incremental version number.
  - `cloudinaryPublicId` (string): Image pointer.
  - `secureUrl` (string): Original source URL.
  - `uploadedAt` (number): Timestamp of upload.
  - `changeNote` (string): Context message written by the designer explaining what changed.

#### B. Feedback Notes (`/posters/{posterId}/comments/{commentId}`)
Timeline comments on design reviews.
* **Fields**:
  - `authorId` (string): UID of commenter.
  - `authorRole` (string): `"designer" | "reviewer" | "admin"`.
  - `text` (string): Comment content.
  - `createdAt` (number): Timestamp.

---

### 5. In-App Notifications (`/notifications/{uid}/items/{notifId}`)
Provides real-time notifications to users.

* **Path**: `/notifications/{uid}/items/{notifId}` (where `{uid}` is the recipient's user ID)
* **Fields**:
  - `type` (string): Message type (`"new_submission" | "approved" | "changes_requested" | "rejected" | "new_comment"`).
  - `posterId` (string): Document ID of the subject poster.
  - `message` (string): The notification message text.
  - `read` (boolean): Flag indicating whether the user read it.
  - `createdAt` (number): Timestamp of delivery.

---

## 3. Database Relationships

- **User to Poster**: A poster is linked to its creator via `uploadedBy` (which matches a User UID).
- **Poster to Category**: A poster points to a category via the `category` string field.
- **Review Actions**: Status changes link the approving agent via the `reviewedBy` field.
- **Notification Routing**: When a change occurs, the system queries the Firestore `/users` database to check preference flags before fanning out copies of a notification document to target users' `/notifications/{uid}/items` inbox.

## 4. Firestore Security Rules Breakdown

All read/write operations attempting to contact Firestore directly from a client browser are validated against our security rules. Let's look at the active [`firebase/firestore.rules`](file:///c:/Users/JOSHUA%20ZAZA/Downloads/kraftdesk/firebase/firestore.rules) file.

### Section-by-Section Explanation

#### Helper functions:
```javascript
function getRole() {
  return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role;
}
```
* **Explanation**: A helper function that inspects the logged-in user's profile document inside Firestore and retrieves their role (`"designer"`, `"reviewer"`, or `"admin"`).

#### Users Collection:
```javascript
match /users/{uid} {
  allow read: if request.auth != null;
  allow create: if request.auth != null && request.auth.uid == uid
    && request.resource.data.role == "designer";
  allow update: if request.auth != null && (
    request.auth.uid == uid && request.resource.data.role == resource.data.role
    || getRole() == "admin"
  );
}
```
* **Security Guardrails**:
  1. Anyone logged in can read user profiles (needed to render commenter names).
  2. Users can only create their own profile (`request.auth.uid == uid`), and they are forced to signup with the default `"designer"` role. They cannot register themselves as admin or reviewer.
  3. When users edit their own profile, they cannot modify their role (`request.resource.data.role == resource.data.role`). Only users who already possess the `"admin"` role can change other users' roles.

#### Categories Collection:
```javascript
match /categories/{categoryId} {
  allow read: if true;
  allow write: if request.auth != null && getRole() == "admin";
}
```
* **Security Guardrails**: Anyone can browse categories, but only logged-in Administrators can write or modify them.

#### Posters Collection & Subcollections:
```javascript
match /posters/{posterId} {
  allow read: if true;
  allow create: if request.auth != null && request.auth.uid == request.resource.data.uploadedBy;
  allow update: if request.auth != null && (
    (
      getRole() in ["reviewer", "admin"]
      && (
        request.resource.data.status == resource.data.status
        || request.resource.data.status in ["approved", "changes_requested", "rejected"]
      )
    )
    || (
      request.auth.uid == resource.data.uploadedBy
      && (
        request.resource.data.status == resource.data.status
        || request.resource.data.status in ["draft", "pending", "changes_requested"]
      )
    )
  );

  match /versions/{versionNumber} {
    allow read: if true;
    allow write: if request.auth != null;
  }

  match /comments/{commentId} {
    allow read: if true;
    allow create: if request.auth != null;
  }
}
```
* **Security Guardrails**:
  1. **Read Gating**: All posters, versions, and comments are readable (`allow read: if true`). This is designed so poster elements can be referenced. The real protection of the high-res file is at the application layer—unauthorized users only get watermarked image links in their browser; the high-resolution clean file is only downloadable by requesting it through a secure server-side API endpoint that verifies ownership or staff role.
  2. **Create**: Designers can only upload posters where they sign the document as the creator (`uploadedBy == uid`).
  3. **Update**:
     - Reviewers/Admins can write reviews (approved, rejected, or changes requested). They *cannot* modify the status directly to "published" from the client—publishing has to go through our Next.js API server to ensure the un-watermarked preview is generated.
     - Designers can update draft posters they own, submit them to "pending", or resubmit them after changes are requested. They cannot change status to approved, rejected, or published.
  4. **Subcollections**: Comments and versions require a logged-in account to create, but are readable by anyone.

#### Notifications:
```javascript
match /notifications/{uid}/items/{notifId} {
  allow read, update: if request.auth != null && request.auth.uid == uid;
  allow create: if request.auth != null;
}
```
* **Security Guardrails**:
  - Users can read and mark notifications read only for their own inbox (`request.auth.uid == uid`).
  - Creation is allowed by any authenticated user. This allows server-side API routes and server actions acting on behalf of commenter/reviewer actions to write items into the receiver's collection.

## 5. Realtime Database Lock (Inert Configuration)

KraftDesk only uses Cloud Firestore. It does **not** use the Firebase Realtime Database. However, to keep the project secure in case the Realtime Database is accidentally enabled on the Firebase console, the project includes the [`firebase/database.rules.json`](file:///c:/Users/JOSHUA%20ZAZA/Downloads/kraftdesk/firebase/database.rules.json) file which locks down all operations:

```json
{
  "rules": {
    ".read": false,
    ".write": false
  }
}
```
This forces all client connection requests to be completely rejected at the Firebase database server gateway.
